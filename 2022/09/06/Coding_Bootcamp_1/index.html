Interviewing

Interviewee:

  • Make sure know problem to solve
  • resolve ambiguities
    • ask for clarifications
  • “Talk out”
    • keep up running commentary of thought process
    • consider multiple choices

WhiteBoard Technique

  • write legibly
  • adapt to size
  • use a corner to write down requirements, example figures
  • leave enough space for code
  • don’t be afraid to erase

Interviewer

  • Give questions in order
  • State questions clearly and concisely, no need to spell every detail
  • Give hints when interviewee got stuck
  • Control the time
    • Don’t start new question when time is limited
    • Speed along the current question

Testing Requires Its Own Mindset

  • When programming, make code work
  • When testing, make code fail
  • Think about how code could fail, and what conditions might cause it fail
    • bad input data
    • external errors

Test a Pen

  • Drop the pen

  • Write on variety surface

  • Normal operation

    • Does it extend smoothly, cap easy to remove
    • does ot generates smooth line on paper

  • Unexpected operation

    • write upside down?
    • white on sth other than paper?

Unit testing

  • Determine smallest testable pert of program
    • function, module, etc.
  • a set of test cases to verify correct outputs for given inputs
    • normal operation
    • boundary conditions
    • erroneous input
  • closely related to defensive programming

Boundary Conditions / Edge Cases

  • Consider the extremes that might be encountered
    • empty input
    • extremely large imput
    • smallest valid value
    • largest valid value
  • inputs that can cause boundary or error conditions in internal functions
    • division by 0, overflow, accessing invalid array index
    • odd number of items in search or divide and conquer algorithms

White vs. Black

  • Depending on how much access you have to the code under test,
    • Black box testing: see only inputs and outputs
    • White boc

Defensive programming

  • a form of defensive design intended to develop programs that are capable of detect potential security abnormalities and make predetermined response.

Never trust the Input

  • Check the validity for every argument you don’t control
  • Check Intermediate Results
    • overflow, underflow, div by 0 ….

The Robustness Principle

In computing, the robustness principle is a design guideline for software that states: “be conservative in what you do, be liberal in what you accept from others”. It is often reworded as: “be conservative in what you send, be liberal in what you accept”

Performance Bottlenecks

  • A program should be considered to have failed if it is too slow even if the result eas corrent
    • e.g. real-time operation
  • Need to consider inputs that could cause too-slow operation
    • large inputs for o(n^3) or O(2^N) algorithms
    • sorted input to QuickSort(>> next lecture)

Multi-Threading

  • Using multiple threads introduces uncertainty
  • Race condition:
  • if the results differ depending on the order in which multiple threads access a variable, we call that a data race
  • Fix in java:
  • volatile keyword
    • disable caching of the variable, so every modification is immediately written to main memorym and every read uses the current value in main memory
    • useful when a single thread is making updates and one or more other threads are reading
  • synchronized keyword
    • enforces one-at-a-time accessing
