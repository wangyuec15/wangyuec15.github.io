Aug 29, 2021

47: 创建团队的项目

可以应用marketplace服务

48: 怎样选择适合自己团队的工作流

考虑的因素：团队人员的组成、研发设计能力、输出产品的特征、项目的难易程度
主干开发：任何人产生的commit及时的融入团队仓库，适用于组件开发的团队，成员能力强，人员少，沟通顺畅，用户升级组件成本低
Git Flow：适用于不具备主干开发能力，有预定的发布周期，需要执行严格的发布流程。适合周期比较长，对质量要求比较高的，相对比较繁琐
GitHub Flow：适用于不具备主干开发能力，随时集成随时发布
GitLab Flow：适用于不具备主干开发能力，无法控制准确的发布时间，但又要求不停的集成（带环境分支：需要通过测试环境的验证才能发布）（带发布分支：软件跟硬件走，需要对外发布和维护不同版本）

49: 如何挑选合适的分支集成策略

项目insights左边的network功能 特性分支
选择集成分支create pull request
create a merge commit：确定merge，紫色图标说明成功了，可以delete branch，不急着。合并到了同一个分支
回退（实际中不要做）
squash and merge：将多个commit放在同一个commit更新分支。在主干上多了一个commit。也可以形成线性的分支，没有merge的节点。
rebase and merge：将多个commit更新到主干上

53: 团队协作时如何做多分支的集成

后融合时 解决冲突 删掉分割线commit merge

先后rebase and merge

55: 怎样把产品包发布到GitHub上

切换到特性分支上 修改Travis config
deploy的配置 api的key
settings-installed GitHub apps 链接
开发者设置personal access tokens 要存储，在Travis设置起来

检查pull request，等待travis构建分支，merge到master分支
看到release说明成功了
可以看到release包的名称，war包，工程包，源代码
